# Пользовательские tools (tools-as-data) — пример `clear_register`

Начиная с v1.1 в `LN_MCP_1C` пользователь может регистрировать собственные MCP-tools как записи справочника `LN_МСРПользовательскиеИнструменты` (через `register_tool` / `update_tool` / `delete_tool` / `list_custom_tools`). После регистрации tool сразу виден в `tools/list` и доступен любому MCP-клиенту (Cursor, Claude Desktop, curl, …).

В коде tool'а доступны переменные:

- `Аргументы` — структура входа (соответствует `input_schema`),
- `Результат` — значение, которое возвращается клиенту (сериализуется в JSON).

## Важное ограничение контекста `Выполнить()`

Код tool'а исполняется через `Выполнить()` на сервере. В этом контексте **bare-менеджеры глобального контекста недоступны** — обращение `РегистрСведений.Имя.СоздатьНаборЗаписей()` падает с `Переменная не определена (РегистрСведений)`. При этом работают:

- `Метаданные`, `Новый Запрос(...)`, глобальные функции (`СтрНачинаетсяС`, `Сред`, …),
- текст запроса `… ИЗ РегистрСведений.Имя` (парсится движком запросов, не BSL-компилятором).

**Обход:** создавать объекты через `Новый(Тип("..."))` вместо обращения к менеджеру. Например, `НаборЗаписей` — через `Новый(Тип("РегистрСведенийНаборЗаписей.Имя"))`.

## Пример: `clear_register`

Tool очищает независимый регистр сведений по имени — одной транзакцией, через запись пустого `НаборЗаписей` без отбора. Возвращает имя регистра и количество оставшихся записей.

**`input_schema`:**

```json
{
  "type": "object",
  "properties": {
    "name": { "type": "string", "description": "Имя регистра сведений, например ВерсииОбъектов (можно с префиксом РегистрСведений.)" }
  },
  "required": ["name"]
}
```

**Код (BSL):**

```bsl
Имя = Аргументы.name;
Если СтрНачинаетсяС(Имя, "РегистрСведений.") Тогда
    Имя = Сред(Имя, СтрДлина("РегистрСведений.") + 1);
КонецЕсли;
ИмяТипа = "РегистрСведенийНаборЗаписей." + Имя;
НЗ = Новый(Тип(ИмяТипа));
НЗ.Записать();
Запрос = Новый Запрос;
Запрос.Текст = "ВЫБРАТЬ КОЛИЧЕСТВО(*) КАК Осталось ИЗ РегистрСведений." + Имя;
Выборка = Запрос.Выполнить().Выбрать();
Выборка.Следующий();
Результат = Новый Структура("Регистр, Осталось", Имя, Выборка.Осталось);
```

Параметр `name` принимает имя регистра без префикса (`ВерсииОбъектов`) или с ним (`РегистрСведений.ВерсииОбъектов`) — префикс отрезается автоматически.

### Ручная регистрация (JSON-RPC)

```powershell
$body = '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"register_tool","arguments":{"name":"clear_register","description":"Очистка регистра сведений по имени.","input_schema":"{\"type\":\"object\",\"properties\":{\"name\":{\"type\":\"string\"}},\"required\":[\"name\"]}","code":"НЗ = Новый(Тип(\"РегистрСведенийНаборЗаписей.\" + Аргументы.name)); НЗ.Записать(); Результат = 0;","safe_mode":false,"active":true}}}'
Invoke-RestMethod -Uri "http://localhost:81/BASE/hs/mcp/rpc" -Method Post `
  -Headers @{ Authorization = "Basic <BASE64>"; "Content-Type" = "application/json; charset=utf-8" } `
  -Body ([Text.Encoding]::UTF8.GetBytes($body))
```

### Скрипт-регистратор: `scripts/register_clear_register.ps1`

Идемпотентный скрипт: проверяет наличие tool'а через `list_custom_tools` и вызывает `update_tool` если tool уже есть, иначе `register_tool`. Сам считает UTF-8 Base64 от `логин:пароль` (важно для кириллических паролей), печатает готовую запись для `.cursor/mcp.json`.

```powershell
# Текущая база
.\scripts\register_clear_register.ps1 -BaseUrl "http://localhost:81/PPKSIP_2" -User IBT -Password "хъ"

# Другая база
.\scripts\register_clear_register.ps1 -BaseUrl "http://localhost:81/OtherBase" -User IBT -Password "secret"

# Свое имя tool'а (чтобы несколько баз не конфликтовали в Cursor)
.\scripts\register_clear_register.ps1 -BaseUrl "http://localhost:81/OtherBase" -User IBT -Password "secret" -ToolName "clear_register_other"
```

> Файл сохранён в UTF-8 with BOM — это обязательно для корректного парсинга кириллицы PowerShell на русской Windows.

## Перенос tool'а в другую базу

Пользовательский tool — это **данные базы** (элемент `Справочник.LN_МСРПользовательскиеИнструменты`), а не объект расширения. В новой базе справочник пуст — tool нужно перерегистрировать.

1. Установить расширение `LN_MCP_1C` и сделать веб-публикацию (`http://localhost:81/<ИМЯБАЗЫ>`).
2. Проверить healthcheck: `GET /hs/mcp/health` → `{"status":"ok"}`.
3. Запустить `scripts/register_clear_register.ps1` с URL/кредами новой базы.
4. В Cursor добавить отдельную запись в `.cursor/mcp.json` (см. `.cursor/mcp.json.example`):

```json
{
  "mcpServers": {
    "ln-mcp-1c-base-a": {
      "url": "http://localhost:81/BaseA/hs/mcp/rpc",
      "transport": "http",
      "headers": { "Authorization": "Basic <BASE64_A>" }
    },
    "ln-mcp-1c-base-b": {
      "url": "http://localhost:81/BaseB/hs/mcp/rpc",
      "transport": "http",
      "headers": { "Authorization": "Basic <BASE64_B>" }
    }
  }
}
```

`<BASE64_…>` = `[Convert]::ToBase64String([Text.Encoding]::UTF8.GetBytes("USER:PASSWORD"))`.

## Ограничения

- Tool **безвозвратно** очищает весь регистр одной транзакцией — использовать осознанно.
- Подходит для **независимых регистров сведений** (непериодических и периодических — пустой набор без отбора чистит все записи). Для **подчинённых регистратору** регистров (накопления / сведений с регистратором) нужен отбор по регистратору — этот tool для них не годится.
- `safe_mode=false` (полный режим записи). В коде нет запрещённых подстрок (`Выполнить(`, `файл`, `comобъект`, …), поэтому даже при `execute_code_default_safe=true` в `LN_МСРНастройки` регистрация и вызов проходят.
- Если в Cursor подключены **несколько баз одновременно** и в обеих tool называется одинаково — имена схлопнутся. Решение: уникальные имена через `-ToolName` либо держать подключённой одну базу за раз.
